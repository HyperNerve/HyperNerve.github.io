Design Patterns

Design patterns are proven solutions to common software design problems. They provide templates for structuring code in a reusable, maintainable, and scalable way.

1. Categories of Design Patterns

  • Creational: Object creation mechanisms
  • Structural: Composition of classes and objects
  • Behavioral: Communication between objects

2. Creational Patterns

Singleton

Ensures only one instance of a class exists.

class Singleton:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

Factory Method

Creates objects without specifying exact class.

class ShapeFactory:
    def create_shape(self, type):
        if type == "circle":
            return Circle()
        elif type == "square":
            return Square()

3. Structural Patterns

Adapter

Allows incompatible interfaces to work together.

class OldPrinter:
    def print_text(self, text):
        print(text)

class PrinterAdapter:
    def __init__(self, old_printer):
        self.old_printer = old_printer
    def print(self, msg):
        self.old_printer.print_text(msg)

Decorator

Adds behavior to objects dynamically.

def bold(func):
    def wrapper():
        return "" + func() + ""
    return wrapper

@bold
def greet():
    return "Hello"

4. Behavioral Patterns

Observer

Notifies dependent objects of state changes.

class Subject:
    def __init__(self):
        self._observers = []
    def attach(self, obs):
        self._observers.append(obs)
    def notify(self, msg):
        for obs in self._observers:
            obs.update(msg)

Strategy

Encapsulates interchangeable algorithms.

class SortStrategy:
    def sort(self, data): pass

class QuickSort(SortStrategy):
    def sort(self, data): return sorted(data)

class Context:
    def __init__(self, strategy):
        self.strategy = strategy
    def execute(self, data):
        return self.strategy.sort(data)

5. Benefits of Design Patterns

  • Reusability: Apply solutions across projects
  • Maintainability: Easier to update and refactor
  • Scalability: Support system growth
  • Communication: Shared vocabulary among developers

6. Commonly Used Patterns

  • MVC (Model-View-Controller): Separates UI, logic, and data
  • Command: Encapsulates requests as objects
  • Facade: Simplifies complex subsystems
  • Proxy: Controls access to objects

7. Anti-Patterns

  • Spaghetti Code: Unstructured and hard to maintain
  • God Object: One class does too much
  • Golden Hammer: Overuse of a familiar pattern

8. Pattern Selection Guidelines

  • Understand the Problem: Identify recurring design issues
  • Match Intent: Choose patterns that align with goals
  • Avoid Overengineering: Use patterns only when needed

9. Tools and Libraries

  • Spring Framework: Uses Singleton, Factory, Proxy
  • Django: Implements MVC as MTV
  • React: Component-based architecture with Observer-like behavior
