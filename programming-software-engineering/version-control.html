Version Control Systems (VCS)

Version Control Systems track changes to files over time, enabling collaboration, rollback, and branching in software development. They are essential for managing codebases and coordinating team efforts.

1. Goals of VCS

  • Track Changes: Record history of modifications
  • Collaboration: Multiple developers work on shared code
  • Branching: Isolate features or fixes
  • Rollback: Revert to previous versions

2. Types of VCS

  • Local VCS: Tracks changes on a single machine
  • Centralized VCS: Single server stores all versions (e.g., SVN)
  • Distributed VCS: Each user has a full copy of the repository (e.g., Git)

3. Git Basics

  • Repository: Project directory tracked by Git
  • Commit: Snapshot of changes with a message
  • Branch: Independent line of development
  • Merge: Combine changes from different branches

4. Common Git Commands

git init           # Initialize repository
git clone URL      # Copy remote repository
git add file       # Stage changes
git commit -m "msg" # Save changes
git push           # Upload to remote
git pull           # Download and merge changes
git checkout branch # Switch branches
git merge branch   # Merge branch into current

5. Branching Strategy Example

Main Branch: production-ready code
Feature Branch: new feature development
Hotfix Branch: urgent bug fixes

6. GitHub Workflow

  • Fork: Copy repository to your account
  • Clone: Download to local machine
  • Commit & Push: Save and upload changes
  • Pull Request: Request to merge changes into original repo

7. Benefits of VCS

  • History: View and revert changes
  • Collaboration: Multiple contributors with conflict resolution
  • Experimentation: Try new ideas safely with branches

8. Conflict Resolution

  • Merge Conflict: Occurs when changes overlap
  • Manual Resolution: Edit conflicting files and commit

9. Tools and Platforms

  • Git: Most popular distributed VCS
  • GitHub: Cloud hosting for Git repositories
  • GitLab: DevOps platform with Git support
  • Bitbucket: Git and Mercurial hosting

10. Best Practices

  • Commit Often: Save work incrementally
  • Write Clear Messages: Describe changes meaningfully
  • Use Branches: Isolate features and fixes
  • Review Before Merge: Use pull requests and code reviews
